1.1 · 概念建立

Windows API 是什麼？

在寫任何一行程式之前，先搞懂「你在和誰說話」。

你的程式不能直接控制電腦的硬體（CPU、記憶體、磁碟）。Windows 作業系統站在中間，負責管理這一切。如果你想做到「列出所有程序」、「終止一個程式」、「讀寫檔案」，你必須透過 Windows 提供的介面來請求。

這個介面就叫做 Windows API（Application Programming Interface，應用程式介面）。它是一大堆 Windows 預先寫好的函數，你呼叫它們，Windows 就幫你做事。

🧠 比喻：餐廳點餐

你（你的程式）→ 服務生（Windows API 函數）→ 廚房（作業系統核心）→ 食材（硬體）

你不能直接走進廚房拿食材。你只能告訴服務生你要什麼。Windows API 就是那份菜單和那位服務生。

例如：GetCurrentProcessId() 就像問服務生「我現在坐在幾號桌？」——Windows 知道，它告訴你。

📦

User Mode

你的程式執行的地方。權限有限，不能直接碰硬體。

⚙️

Kernel Mode

Windows 核心執行的地方。有最高權限，管理一切資源。

📞

System Call

你的程式呼叫 API 時，Windows 在背後做的「跨越邊界」動作。

📚

Win32 API

Windows API 的正式名稱。數千個函數，涵蓋程序、檔案、網路、UI 等一切。

💡 關鍵理解
C++ 本身沒有「列出所有程序」的功能。這件事由 Windows 提供。你寫 C++ 只是為了能夠呼叫 Windows 提供的函數，然後控制系統。這就是為什麼防毒軟體、防火牆幾乎都用 C++ 寫——因為 C++ 可以直接使用這些底層函數。
1.2 · 程式結構

標頭檔與函式庫

#include 那幾行到底在引入什麼？每一個都有具體的用途。

C++ 最常見的 include 組合 
#include <windows.h>
#include <tlhelp32.h>
#include <iostream>
#include <string>

#pragma comment(lib, "advapi32.lib")
<windows.h>

Windows API 的總開關。引入這一個，就同時引入了幾百個 Windows 函數的宣告，包括 OpenProcess、CreateFile、HANDLE、DWORD 等型別定義。

幾乎每個 Windows 系統程式都需要它。

<tlhelp32.h>

Tool Help Library 的縮寫。提供列舉程序、執行緒、模組的函數，例如 CreateToolhelp32Snapshot、Process32First。

「拍一張系統快照，然後一個個讀」——這個概念就來自這個標頭。

<iostream>

標準 C++ 的輸入輸出。讓你用 cout 印出文字、cin 讀使用者輸入。

這個你已經懂，不需要解釋。

#pragma comment

#pragma comment(lib, "xxx.lib") 是告訴連結器（Linker）：「請把這個 .lib 函式庫的實作一起打包進來。」

advapi32.lib 包含登錄檔操作（RegOpenKeyEx 等）的實作。不加的話，程式會編譯成功但連結失敗。

🧠 .h 標頭檔 vs .lib 函式庫的差別

.h 標頭檔：只是「菜單」——告訴你有哪些函數、長什麼樣子（函數名稱、參數、回傳型別）。

.lib 函式庫：真正的「廚房」——函數的實際程式碼在這裡。你的程式要跑起來，必須把實作也連結進去。

比喻：.h 是說明書，.lib 是真正的零件。兩者缺一不可。

標頭 / 函式庫用途分類常用函數
windows.h核心 Windows APIOpenProcess, CreateFile, GetLastError…
tlhelp32.h程序/執行緒列舉CreateToolhelp32Snapshot, Process32First…
winsvc.hWindows 服務OpenSCManager, CreateService…
iphlpapi.h + .lib網路連線資訊GetExtendedTcpTable…
psapi.h + .lib程序記憶體/模組EnumProcessModules, GetModuleFileNameEx…
advapi32.lib登錄檔、安全性、服務RegOpenKeyEx, AdjustTokenPrivileges…
⚠️ 常見錯誤
如果你看到連結錯誤 unresolved external symbol，通常表示你有 #include 對應標頭，但忘了加 #pragma comment(lib, "xxx.lib")。
1.3 · 核心概念

HANDLE：控制代碼

Windows 程式裡最常出現的型別。不理解它，後面什麼都看不懂。

HANDLE 是 Windows 給你的一個「憑證編號」。當你要求 Windows 開啟一個資源（程序、檔案、登錄機碼、執行緒…），Windows 不會直接把資源交給你——它會給你一個號碼牌，你之後憑這個號碼牌來操作那個資源。

🧠 比喻：行李寄存

你把行李（資源）交給寄存處（Windows），寄存處給你一個號碼牌（HANDLE）。你之後拿號碼牌來領取、操作行李。你不直接碰行李本身。

HANDLE hProcess = OpenProcess(...);
← 你去「寄存處」說「我要操作 PID=1234 的程序」，Windows 給你號碼牌 hProcess。

C++ HANDLE 的使用流程 
// 1. 取得 HANDLE（向 Windows 要號碼牌）
HANDLE hProcess = OpenProcess(
    PROCESS_TERMINATE,  // 我要做什麼
    FALSE,              // 子程序要不要繼承
    1234                // 哪個程序的 PID
);

// 2. 確認成功（號碼牌是否有效）
if (hProcess == nullptr) {
    // 失敗：Windows 不給你這個號碼牌
}

// 3. 用 HANDLE 操作資源
TerminateProcess(hProcess, 0);

// 4. 用完一定要還（釋放 HANDLE）
CloseHandle(hProcess);
// 不還會造成「HANDLE 洩漏」，
// 系統資源慢慢被吃光
HANDLE hProcess

HANDLE 是 Windows 定義的型別，本質上是一個整數（指標大小），代表一個資源的「憑證」。hProcess 是變數名稱，習慣上前綴 h 代表 Handle。

PROCESS_TERMINATE

你在申請這個 HANDLE 時，必須說明你打算用它做什麼。Windows 會根據你說的用途決定是否允許。

PROCESS_TERMINATE = 我只要終止這個程序的權限。
PROCESS_ALL_ACCESS = 我要所有權限（需要更高身份）。

== nullptr

如果 OpenProcess 失敗，它回傳 nullptr（空指標，也就是 0）。這表示 Windows 拒絕給你號碼牌——可能因為程序不存在，或你權限不足。

有些函數失敗時回傳 INVALID_HANDLE_VALUE（= -1）而非 nullptr，要看文件確認。

CloseHandle

歸還號碼牌。Windows 釋放與這個 HANDLE 關聯的內部資源。每個成功取得的 HANDLE 都必須呼叫 CloseHandle。

忘記關的話，程式跑久了系統資源會洩漏。這是 Windows 程式設計最常見的 bug 之一。

✅ 固定流程
Windows API 幾乎都遵循：取得 HANDLE → 檢查是否成功 → 使用 → CloseHandle。記住這個模式，你就能看懂 90% 的 Windows 程式。
1.4 · 型別系統

DWORD、BOOL、LPSTR…Windows 的型別語言

Windows API 不用 int、char，而是用自己定義的型別名稱。搞懂對應關係就不再陌生。

你會看到大量全大寫的型別，像 DWORD、HANDLE、BOOL、LPSTR。這些不是新的東西，它們只是 C++ 基本型別的別名（用 typedef 定義），這樣做是為了讓程式碼在不同平台上容易移植，也讓名稱更有語意。

Windows 型別等同於意思與用途
DWORDunsigned long（32位元）最常用的整數型別。PID、錯誤碼、大小、旗標都用它。
BOOLintWindows 的布林值。TRUE=1，FALSE=0。注意：不是 C++ 的 bool！
HANDLEvoid*（指標）資源的「號碼牌」。前節已詳細說明。
LPSTRchar*LP = Long Pointer，STR = String。指向 ANSI 字元字串的指標。
LPWSTRwchar_t*W = Wide（寬字元，Unicode）。指向 Unicode 字串的指標。
LPCSTRconst char*C = Const。指向唯讀 ANSI 字串的指標。
LPCWSTRconst wchar_t*指向唯讀 Unicode 字串的指標。
BYTEunsigned char8 位元無號整數。常用在緩衝區（buffer）。
WORDunsigned short（16位元）16 位元無號整數。
VOIDvoid沒有意義的型別，等同 C++ 的 void。
WINAPI__stdcall呼叫慣例。告訴編譯器這個函數的參數傳遞方式。大多數 Windows API 都用它。
C++ 字串命名慣例：A 版本 vs W 版本 
// A 版本：ANSI 字串（char*），舊式
char name[256];
DWORD size = 256;
GetComputerNameA(name, &size);

// W 版本：Unicode 字串（wchar_t*），現代 Windows 使用
wchar_t nameW[256];
DWORD sizeW = 256;
GetComputerNameW(nameW, &sizeW);

// 印出 Unicode 字串要用 wcout
using namespace std;
wcout << nameW << endl;

// L"..." 前綴：把字串標記為 Unicode
LPCWSTR path = L"C:\\Windows\\System32";
//             ↑ L 前綴讓這個字串變成 wchar_t[]
函數名尾A / 尾W

Windows 很多函數有兩個版本：FunctionA（ANSI）和 FunctionW（Wide/Unicode）。現代程式應盡量用 W 版本，因為 Unicode 才能正確處理中文、日文等非英文字符。

L"..."

字串前面的 L 是字面量前綴。L"hello" 的型別是 wchar_t[]，每個字符佔 2 bytes（Unicode）。沒有 L 的 "hello" 是 char[]，每個字符 1 byte（ANSI）。

&size（傳址）

你把 size 的地址傳給函數，讓函數直接修改你的變數。函數用完後，size 的值會被更新為實際的長度。

這是 Windows API 常見的「輸出參數」模式：函數透過指標回傳多個值。

🔮 記住這個規律
A = ANSI（舊，char*），W = Wide（新，wchar_t*）。看到函數名字尾是 A 或 W，你就知道它用哪種字串。中文 Windows 環境下，幾乎所有路徑、名稱都應該用 W 版本，否則中文路徑會出錯。
1.5 · 實戰解析

第一支程式：逐行全解析

取得電腦名稱與記憶體資訊。每一行、每一個參數都說清楚。

C++ system_info.cpp — 全程式逐行解析 
#include <windows.h>
#include <iostream>
using namespace std;

int main() {

  // ── 取得電腦名稱 ──
  char compName[MAX_COMPUTERNAME_LENGTH + 1];
  DWORD nameSize = sizeof(compName);

  BOOL ok = GetComputerNameA(
      compName,
      &nameSize
  );

  if (ok) {
      cout << "電腦名稱: "
           << compName << endl;
  }

  // ── 取得記憶體資訊 ──
  MEMORYSTATUSEX mem;
  mem.dwLength = sizeof(mem);

  GlobalMemoryStatusEx(&mem);

  cout << "記憶體使用率: "
       << mem.dwMemoryLoad
       << "%" << endl;

  cout << "總記憶體: "
       << (mem.ullTotalPhys
            / 1024 / 1024)
       << " MB" << endl;

  return 0;
}
char compName[MAX_COMPUTERNAME_LENGTH + 1]

宣告一個字元陣列（緩衝區），用來接收 Windows 回傳的電腦名稱字串。

MAX_COMPUTERNAME_LENGTH 是 Windows 定義的常數 = 15（最長電腦名稱）。+1 是為了放字串結尾的 \0（null terminator）。

DWORD nameSize = sizeof(compName)

這個變數有兩個用途：呼叫前告訴函數「我的緩衝區有多大」，呼叫後函數會把它改成「實際寫入了幾個字元」。

sizeof(compName) 回傳整個陣列的 byte 數 = 16。

GetComputerNameA(compName, &nameSize)

呼叫 Windows API，讓 Windows 把電腦名稱寫進 compName 這塊記憶體。

第一個參數：要寫入的緩衝區位址。
第二個參數：&nameSize = 傳「nameSize 的地址」，讓函數可以修改它的值。
回傳值：成功=TRUE，失敗=FALSE。

MEMORYSTATUSEX mem

MEMORYSTATUSEX 是 Windows 定義的 struct（結構），裡面裝了記憶體相關的各種數值。你宣告一個這種型別的變數 mem，函數會填滿它的欄位。

mem.dwLength = sizeof(mem)

這是 Windows API 的一個必要儀式：在呼叫函數前，必須把 struct 的 dwLength 欄位設成這個 struct 的大小。Windows 用這個值來確認你傳進來的 struct 版本正確。

忘記設這行→函數回傳失敗。這是初學者最常踩的坑之一。

GlobalMemoryStatusEx(&mem)

讓 Windows 把記憶體資訊填寫到 mem struct 裡。&mem 是 mem 的地址，Windows 拿到地址才能修改它的內容。

mem.dwMemoryLoad

函數執行完後，mem 的各欄位都被填好了。dwMemoryLoad 是記憶體使用百分比（0~100）。其他欄位如 ullTotalPhys（總記憶體 bytes）、ullAvailPhys（可用記憶體 bytes）等。

/ 1024 / 1024

ullTotalPhys 的單位是 bytes。除以 1024 得到 KB，再除一次得到 MB。

1024 bytes = 1 KB，1024 KB = 1 MB。

✅ 你剛剛學到的模式
準備緩衝區（或 struct）→ 設定大小欄位 → 呼叫 API（傳入緩衝區地址）→ API 填好資料 → 你讀取結果。這個模式在 Windows API 裡反覆出現，幾乎每個「取得資訊」的函數都這樣用。
1.6 · 錯誤處理

GetLastError：API 失敗時怎麼辦

Windows API 幾乎都不會直接告訴你哪裡錯了。你需要主動去問。

大部分 Windows API 失敗時，只是回傳 FALSE、nullptr 或 INVALID_HANDLE_VALUE，不會直接說原因。你必須立刻呼叫 GetLastError() 來取得錯誤碼，再把錯誤碼轉成可讀的文字。

C++ 標準錯誤處理流程（完整解析） 
#include <windows.h>
#include <iostream>
#include <string>
using namespace std;

string WinErrMsg(DWORD code) {
    LPSTR buf = nullptr;

    FormatMessageA(
        FORMAT_MESSAGE_ALLOCATE_BUFFER
        | FORMAT_MESSAGE_FROM_SYSTEM
        | FORMAT_MESSAGE_IGNORE_INSERTS,
        nullptr,
        code,
        MAKELANGID(LANG_NEUTRAL,
                   SUBLANG_DEFAULT),
        (LPSTR)&buf,
        0,
        nullptr
    );

    string msg = buf ? buf : "未知錯誤";
    LocalFree(buf);
    return msg;
}

int main() {
    // 故意用不存在的 PID 觸發錯誤
    HANDLE h = OpenProcess(
        PROCESS_ALL_ACCESS,
        FALSE,
        99999   // 不存在的 PID
    );

    if (h == nullptr) {
        DWORD err = GetLastError();
        cout << "失敗！錯誤碼: " << err
             << "\n說明: "
             << WinErrMsg(err) << endl;
    }
    return 0;
}
GetLastError()

回傳最近一次 Windows API 失敗的錯誤碼（DWORD）。必須在失敗後立刻呼叫，因為下一個 API 呼叫可能會覆蓋這個值。

常見錯誤碼：5 = 拒絕存取、2 = 找不到檔案、87 = 參數錯誤、1168 = 找不到元素。

FormatMessageA(...)

把數字錯誤碼轉成人類可讀的文字。這個函數很複雜，參數很多，逐一解析如下：

FORMAT_MESSAGE_ALLOCATE_BUFFER

讓 Windows 自動幫你分配接收訊息的記憶體。你不用自己算緩衝區大小。

FORMAT_MESSAGE_FROM_SYSTEM

指定錯誤訊息的來源是 Windows 系統本身（不是你自己的訊息資源）。

FORMAT_MESSAGE_IGNORE_INSERTS

忽略訊息裡的格式化佔位符（%1、%2 等），直接輸出原始文字，避免崩潰。

nullptr（第二個參數）

訊息來源的模組。因為我們從系統找，所以填 nullptr。

code（第三個參數）

你要查詢的錯誤碼，也就是 GetLastError() 回傳的值。

MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT)

指定訊息語言。LANG_NEUTRAL + SUBLANG_DEFAULT = 用系統預設語言（你的 Windows 是中文就給中文錯誤訊息）。

(LPSTR)&buf

因為我們用了 ALLOCATE_BUFFER 旗標，函數需要一個指向指標的指標（char**）。&buf 是 buf 的地址，函數把分配好的記憶體地址寫進 buf。強制轉型 (LPSTR) 是為了符合參數型別。

LocalFree(buf)

因為是 Windows 幫我們分配的記憶體，必須用 LocalFree 釋放（不是 delete 或 free）。不釋放會造成記憶體洩漏。

⚠️ 重要：立刻呼叫
GetLastError() 必須在 API 失敗後的下一行就呼叫。如果你中間又呼叫了任何其他 Windows 函數（包括 cout），錯誤碼可能已經被覆蓋了。
1.7 · 權限系統

權限與提升：為何需要「以系統管理員執行」

防禦程式最常遇到的障礙——理解 Windows 權限模型，才能知道為什麼某些操作會失敗。

Windows 每個程式執行時，都帶著一個 Token（存取令牌），裡面記錄了這個程式的身份和權限。想對其他程序動手術（終止、讀取記憶體），必須有足夠的權限。

🧠 比喻：醫院識別證

一般員工（標準使用者）→ 只能進一般區域。
外科醫生（系統管理員）→ 可以進手術室。
院長（SYSTEM 帳號）→ 可以進任何地方。

病毒通常想取得「院長」等級的權限，讓自己不可碰觸。你的防禦程式至少需要「外科醫生」等級。

SeDebugPrivilege = 特殊通行證，讓你可以進入其他程序的記憶體空間，即使對方是院長開的診間。

C++ 啟用 SeDebugPrivilege（逐行解析） 
#include <windows.h>
#include <iostream>
using namespace std;

bool EnablePrivilege(
    const wchar_t* privName
) {
    HANDLE hToken = nullptr;

    BOOL ok = OpenProcessToken(
        GetCurrentProcess(),
        TOKEN_ADJUST_PRIVILEGES
            | TOKEN_QUERY,
        &hToken
    );
    if (!ok) return false;

    LUID luid;
    LookupPrivilegeValue(
        nullptr,
        privName,
        &luid
    );

    TOKEN_PRIVILEGES tp = {};
    tp.PrivilegeCount = 1;
    tp.Privileges[0].Luid = luid;
    tp.Privileges[0].Attributes =
        SE_PRIVILEGE_ENABLED;

    AdjustTokenPrivileges(
        hToken, FALSE,
        &tp, sizeof(tp),
        nullptr, nullptr
    );

    DWORD err = GetLastError();
    CloseHandle(hToken);
    return err == ERROR_SUCCESS;
}

int main() {
    if (EnablePrivilege(SE_DEBUG_NAME)) {
        cout << "SeDebugPrivilege 已啟用\n";
    } else {
        cout << "失敗（請以系統管理員執行）\n";
    }
    return 0;
}
OpenProcessToken

取得「自己程式的 Token」的 HANDLE。Token 是記錄你程式身份與權限的物件。

第一參數 GetCurrentProcess()：取得自身程序的 Handle（一個固定的偽控制代碼，不需關閉）。
第二參數 TOKEN_ADJUST_PRIVILEGES | TOKEN_QUERY：我要「修改權限」和「查詢」這個 Token 的能力。
第三參數 &hToken：輸出，函數把 Token 的 Handle 寫進這裡。

LUID luid

LUID = Locally Unique Identifier（本機唯一識別碼）。Windows 用 LUID 來識別每種權限。每台電腦上同一種權限的 LUID 可能不同，所以不能硬編碼，要查詢。

LookupPrivilegeValue(nullptr, privName, &luid)

查詢「這個權限名稱（字串）」對應的 LUID 數字。
第一參數 nullptr：在本機查詢（不是遠端電腦）。
第二參數 privName：權限的名稱字串，例如 SE_DEBUG_NAME = L"SeDebugPrivilege"。
第三參數 &luid：查詢結果寫進 luid。

TOKEN_PRIVILEGES tp

一個 struct，描述「我要修改哪些權限、改成什麼狀態」。
PrivilegeCount = 1：我只改一個權限。
Luid：填入剛才查到的 LUID。
Attributes = SE_PRIVILEGE_ENABLED：把這個權限設為「啟用」狀態。

AdjustTokenPrivileges

實際修改 Token 的權限。第一參數 hToken：要修改的 Token。第二參數 FALSE：不是要停用所有權限（只改指定的）。第三參數 &tp：要做的修改內容。

特別注意：這個函數就算成功回傳 TRUE，也可能實際上沒有修改成功（例如你根本沒有這個權限）。必須再呼叫 GetLastError() 確認是 ERROR_SUCCESS 才算真正成功。

SE_DEBUG_NAME

Windows 預定義的常數字串 = L"SeDebugPrivilege"。擁有這個權限後，你可以開啟系統中任何程序（包括受保護的）的 HANDLE，即使你不是 SYSTEM 帳號。這是對抗「受保護病毒程序」的關鍵權限。

🔴 必須以系統管理員執行
AdjustTokenPrivileges 啟用 SeDebugPrivilege 需要你的程式已經以系統管理員身份執行。如果不是，函數會靜默失敗。在 Visual Studio 中：專案屬性 → 連結器 → 資訊清單檔案 → UAC 執行層級 → requireAdministrator。

Q1 問答

你的 C++ 程式想要「終止另一個程序」，可以直接命令 CPU 停掉它嗎？如果不行，正確的做法是什麼？

📖 參考答案

不行。程式執行在 User Mode，沒有直接控制硬體的權限。正確做法是透過 Windows API 請求 Windows 幫你做：呼叫 TerminateProcess()，把你要終止的程序 HANDLE 傳給它，Windows 核心（Kernel）才是真正執行終止動作的人。

這就像你不能走進廚房自己開爐火，你得告訴服務生（API），由廚房（核心）來做。

B <tlhelp32.h>

📖 解析

<tlhelp32.h> 是 Tool Help Library，專門提供「拍系統快照、列舉程序/執行緒/模組」的工具。<windows.h> 是總開關，但不含這些列舉函數。

Q3 單選

以下程式碼有一個嚴重問題，它是什麼？
HANDLE h = OpenProcess(PROCESS_TERMINATE, FALSE, 1234); TerminateProcess(h, 0);

A PROCESS_TERMINATE 權限不夠，應該用 PROCESS_ALL_ACCESS
B 第二個參數應該是 TRUE
C 沒有呼叫 CloseHandle，造成 HANDLE 洩漏
D TerminateProcess 的第二個參數應該是 1
📖 解析

缺少 CloseHandle(h)。每個成功取得的 HANDLE 都必須在用完後歸還給 Windows，否則系統內部會一直保留這個資源，程式長時間執行後會耗盡系統的 HANDLE 配額。

選項 A 是錯的：PROCESS_TERMINATE 對於終止程序已經足夠，沒必要要求所有權限。選項 D 也是錯的：第二個參數是退出碼，0 或 1 都可以，不影響功能。

Q4 問答

DWORD 和 BOOL 本質上是什麼 C++ 型別？為什麼 Windows 要重新命名它們？

📖 參考答案

DWORD = unsigned long（32位元無號整數）。BOOL = int（不是 C++ 的 bool！）。

Windows 重新命名是為了可移植性（跨不同平台、編譯器時，int 的大小可能不同）和語意清晰（看到 DWORD 就知道這是 Windows 特定的 32 位元值）。

Q5 單選

你要存一個指向 Unicode 字串的指標，且該字串不可被修改，應該用哪個型別？

A LPSTR
B LPWSTR
C LPCWSTR
D LPCSTR
📖 解析

拆解型別名稱：LP = Long Pointer（指標），C = Const（唯讀），W = Wide（Unicode），STR = String。

所以 LPCWSTR = 指向唯讀 Unicode 字串的指標 = const wchar_t*。

LPWSTR（B）雖然是 Unicode，但沒有 Const，字串可以被修改。LPCSTR（D）是唯讀但 ANSI，不是 Unicode。

Q6 問答

為什麼 GetComputerNameA(compName, &nameSize) 要傳 &nameSize 而不是直接傳 nameSize？

📖 參考答案

因為這個函數需要雙向溝通：

呼叫前：你用 nameSize 告訴函數「我的緩衝區有多大」。

呼叫後：函數把 nameSize 改成「實際寫入了幾個字元」。

如果只傳值（nameSize），函數拿到的只是一個複製品，修改了也不影響你的原始變數。傳地址（&nameSize）讓函數直接修改你的變數。這是 C/C++ 的「輸出參數（out parameter）」模式。

Q10 問答

為什麼對抗病毒的防禦程式需要啟用 SeDebugPrivilege？沒有它會怎樣？

📖 參考答案

病毒通常會讓自身程序以高權限（例如 SYSTEM 帳號）執行，使得一般管理員程式無法開啟它的 HANDLE（OpenProcess 回傳 nullptr，錯誤碼 5 = 拒絕存取）。

SeDebugPrivilege（除錯權限）是一個特殊通行證，擁有它的程式可以開啟系統中幾乎任何程序的 HANDLE，繞過正常的存取控制。

沒有它：OpenProcess(PROCESS_TERMINATE, FALSE, virusPid) 很可能失敗，你就無法終止受保護的病毒程序。

Q12 單選

AdjustTokenPrivileges 回傳了 TRUE，代表 SeDebugPrivilege 一定成功啟用了嗎？

A 是，TRUE 就代表成功
B 不一定，還需要呼叫 GetLastError() 確認是 ERROR_SUCCESS
